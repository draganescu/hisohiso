how it works

The channel
has three gates.

Hisohiso is built around a simple security shape: something you have, something you know, and something that identifies how you are asking to enter. For human rooms, the last two can stay optional. For agent rooms, they are part of the pairing flow.

The privacy tradeoff.

Popular messengers give you accounts, discovery, invitations, forwarding, groups, and cross-device history. Hisohiso gives up most of that on purpose. The rough edges are what keep the channel simple: fewer identifiers, fewer social features, and less metadata to build a picture from.

The three gates.

The implementation is deliberately ordinary: a channel can require the link you have, the password you know, and the knock message that says who you are. The server routes the attempt, but it does not get the secret material needed to read the room.

Something you have

The room link

The room secret lives after the # in the URL. Browsers keep that fragment local, so the server sees a channel hash, not the key material.

Something you know

The room password

A room can add a password. Message and knock keys are derived from the room secret plus that password, so the link alone is not enough.

Something you are

The knock message

The knock is the applicant's introduction. Humans can use it as social proof. Agent rooms make it a hidden session secret that the phone must type correctly before auto-approval.

For normal human rooms, the password and knock message are optional. For agent pairing, the CLI prints a fresh pairing code and asks for a hidden knock message, then only approves a phone that has the link, the code, and the knock body.

The link carries the first secret.

Your browser makes a secret when the channel is created. That secret sits after the # in the URL. Browsers do not send that part to the server, so the server gets a channel hash, not the channel key.

Messages are locked before they leave.

When you send a message, your browser encrypts it with a key made from the channel secret and, when present, the room password. The server receives the encrypted blob and sends it to other people or to your agent bridge. Their browser, or the local bridge on your machine, uses the same inputs to open it.

The current implementation uses AES-256-GCM for messages and HKDF-SHA256 to derive separate message and knock keys from the channel secret plus the optional password.

Joining is a knock, not an account.

A new person opens the link and knocks. Someone already inside sees the knock message and approves them. For agents, the bridge joins the channel from your machine, checks that the encrypted knock decrypts with the pairing code, then checks that the knock text matches the hidden session message before it auto-approves. There is no global identity system to search, follow, invite, or leak.

  • No central owner. Anyone in the channel can approve a knock or close the channel.
  • No recovery account. If you lose the link and your local data, there is nothing for the server to recover.
  • No public lobby. A channel is only useful to people or agents that already have the link.

What gets stored, and where?

The main server database stores channel hashes, participant token hashes, and timestamps. Your readable message history is in IndexedDB on your own device.

Server

Routing data

Enough information to know which encrypted channel event belongs where. Not the text of the message.

Your device

Local history

The messages you have received. Clear browser storage and that local copy is gone.

Channels can opt into encrypted offline catch-up. That stores ciphertext only and is capped by retention rules: the newest messages only, with old ciphertext expired instead of becoming permanent cloud history.

What this does not solve.

Hisohiso protects message contents from the server and avoids the identity machinery of social messengers. It does not make a bad link-sharing habit safe, hide your network metadata from your infrastructure, or decide whether an agent command is safe to run. Those choices still belong to you.

  • The web app must still be trusted. End-to-end crypto runs in the client you load. A compromised origin could serve malicious JavaScript, so self-hosting and reading the code matter for high-trust rooms.
  • Metadata does not disappear. The router still needs routing data, timestamps, and network traffic to move encrypted events.
  • Agent safety is separate. A private channel does not prove that a proposed file edit, shell command, or git operation is correct.

If a channel link gets shared too widely, close the channel and make a new one. That is the clean escape hatch.

Readable enough to audit.

The repo is small on purpose. You can read the server, the client crypto, and the deployment files without signing an NDA.
